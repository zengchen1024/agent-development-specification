# 安全规范

> 本文件定义团队级安全开发规范，所有项目必须遵守。
> 安全问题应尽早发现（Shift Left），而非在上线后修复。

---

## 1. 输入验证

### 1.1 验证原则

- **永不信任外部输入**：所有来自用户、第三方 API、消息队列的数据必须验证
- 在系统边界处（HTTP handler、队列消费者）验证，不在业务层重复验证
- 使用白名单验证（允许的值），而非黑名单（禁止的值）

### 1.2 必须验证的场景

- HTTP 请求参数（Query String、Path Params、Request Body）
- 文件上传（类型、大小、内容）
- 配置文件和环境变量（启动时校验）
- 数据库写入前的业务规则验证

### 1.3 SQL 注入防护

- 禁止使用字符串拼接构造 SQL 语句
- 必须使用参数化查询（Prepared Statement）或 ORM
- 定期使用 `sqlmap` 或 SAST 工具扫描

## 2. 认证与授权

### 2.1 认证

> [团队填写] 定义认证方式，例如：
>
> - 使用 JWT 或 Session Cookie（选择其一并统一）
> - Token 有效期：Access Token 15 分钟，Refresh Token 7 天
> - JWT 签名算法使用 RS256（非对称），禁止使用 `none` 算法

### 2.2 授权

- 每个 API 端点必须明确定义所需权限
- 授权检查在 middleware 层统一执行，不分散在业务逻辑中
- 遵循最小权限原则（Principle of Least Privilege）
- 水平越权防护：访问资源时验证资源属于当前用户

### 2.3 密码处理

- 禁止明文存储密码，使用 bcrypt（cost≥12）或 Argon2id
- 禁止将密码记录到日志
- 密码重置使用有时效的一次性 Token（有效期不超过 1 小时）

## 3. 密钥与敏感信息管理

### 3.1 禁止事项（强制）

- 禁止将任何密钥、密码、Token 提交到 Git 仓库（包括测试环境）
- 禁止在日志中输出密钥、完整 Token、数据库连接串
- 禁止在错误响应中返回敏感信息（数据库错误、内部路径等）

### 3.2 密钥存储

> [团队填写] 例如：
>
> - 生产密钥使用 HashiCorp Vault / AWS Secrets Manager 管理
> - 本地开发使用 `.env.local` 文件（加入 `.gitignore`）
> - CI/CD 使用平台提供的 Secret 功能（如 GitHub Secrets）

### 3.3 Git 防护

- 在 pre-commit hook 中集成 `gitleaks` 或 `truffleHog` 扫描
- 如果密钥意外提交，立即轮换密钥（历史记录中的密钥视为泄露）

## 4. 常见漏洞防护

### 4.1 XSS（跨站脚本）

- 所有用户输入在输出到 HTML 前必须转义
- Content-Security-Policy (CSP) 头部设置，禁止 inline script
- 使用框架提供的模板引擎（自动转义），禁止手动拼接 HTML

### 4.2 CSRF（跨站请求伪造）

> [团队填写] 例如：
>
> - 使用 CSRF Token（Synchronizer Token 模式）
> - 或使用 SameSite=Strict Cookie 策略
> - 纯 API 服务（Token 认证）可豁免

### 4.3 SSRF（服务器端请求伪造）

- 禁止直接使用用户提供的 URL 发起服务器端请求
- 如需访问外部 URL，必须白名单验证域名
- 内网 IP 范围（10.x、172.16.x、192.168.x、localhost）必须被拦截

### 4.4 依赖安全

> [团队填写] 例如：
>
> - CI 中集成 `trivy` / `npm audit` / `safety`（Python）定期扫描
> - Critical / High 级别漏洞必须在 7 天内修复
> - 使用 Dependabot 或 Renovate 自动更新依赖

## 5. 数据安全

### 5.1 数据分级

> [团队填写] 定义数据敏感级别，例如：
> | 级别 | 示例 | 处理要求 |
> |------|------|----------|
> | L1 公开 | 产品名称、公开文档 | 无特殊要求 |
> | L2 内部 | 用户 ID、订单号 | 访问控制 |
> | L3 敏感 | 手机号、邮箱、地址 | 加密存储、脱敏展示 |
> | L4 高敏 | 密码、支付信息、身份证 | 加密存储、严格访问审计 |

### 5.2 数据脱敏

- L3/L4 数据在日志和 API 响应中必须脱敏（手机号中间 4 位打码等）
- 测试环境使用脱敏数据，禁止从生产导入真实用户数据

### 5.3 传输安全

- 所有外部通信使用 HTTPS（TLS 1.2+）
- 内部服务间通信根据安全级别决定是否启用 mTLS

## 6. 安全审查流程

### 6.1 Code Review 安全检查

每次 PR Review 必须检查：

- [ ] 是否有新的外部输入未经验证
- [ ] 是否新增了硬编码的密钥或敏感配置
- [ ] 是否有 SQL 拼接或不安全的反序列化
- [ ] 新增的权限检查是否正确
- [ ] 日志是否泄露敏感信息

### 6.2 安全事件响应

> [团队填写] 定义安全事件响应流程，例如：
>
> - 发现安全漏洞立即通过加密渠道上报（security@team.com），禁止在公开 Issue 中描述
> - 高危漏洞响应 SLA：24 小时内确认，72 小时内修复
> - 安全事件复盘报告模板见 `docs/templates/security-incident-report.md`
