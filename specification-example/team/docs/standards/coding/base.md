# 编码规范 · 通用基础

> 本文件定义语言无关的通用编码规范，适用于所有项目和技术栈。
> 各语言专项规范（go.md / python.md / typescript.md）在此基础上补充语言特有约定。

---

## 1. 命名规范

### 1.1 通用原则

- **表意优先**：名称应清晰表达意图，避免缩写（常见业务缩写除外，需在项目 glossary 中定义）
- **一致性**：同一概念在整个代码库中使用同一名称
- **避免歧义**：不使用 `data`、`info`、`util`、`helper` 等含义模糊的名称作为模块/包名

### 1.2 命名长度原则

- 变量作用域越大，名称应越具体
- 局部变量在上下文清晰时可缩短（如循环变量 `i`、`err`）
- 函数/方法名应能自描述行为，无需注释补充

### 1.3 布尔值命名

> 布尔变量和返回布尔的函数，名称应能形成肯定语句：
>
> - 变量：`isEnabled`、`hasPermission`、`canRetry`
> - 函数：`IsValid()`、`HasChildren()`

### 1.4 禁止事项

- 禁止使用单字母变量名（`i`、`j` 用于循环索引除外）
- 禁止使用负向命名（`notFound`、`disabled` 在逻辑运算中易造成双重否定）
- 禁止使用语言关键字或标准库名称作为变量名

## 2. 文件与目录组织

### 2.1 目录结构原则

- 按业务域组织，而非按技术层（如 controllers、models）组织
- 每个目录应有单一职责，目录名使用小写，多词用 `-` 连接（kebab-case）
- 避免过深的目录嵌套（超过 4 层应考虑重组）

### 2.2 文件大小

> [团队填写] 建议设置参考上限，例如：
>
> - 单文件不超过 500 行（包含注释）
> - 超过上限时，考虑拆分为多个文件或提取子模块

### 2.3 文件命名

- 文件名全小写，使用 `_`（snake_case）或 `-`（kebab-case）分词（按语言习惯，各语言规范中指定）
- 测试文件与被测文件同目录，命名规则参见测试规范

## 3. 函数与方法

### 3.1 单一职责

- 每个函数只做一件事，函数名能准确描述这件事
- 函数行数建议不超过 50 行；超过时检查是否可拆分（而非强制限制）

### 3.2 参数设计

- 参数数量建议不超过 4 个；超过时使用结构体/对象传参
- 避免使用布尔参数控制函数行为（拆为两个函数更清晰）
- 避免使用 `nil` / `null` 作为业务语义的信号量

### 3.3 返回值

- 优先使用返回值传递错误，而非抛出异常（具体见语言规范）
- 函数不应既修改状态又返回值（Command Query Separation 原则）
- 避免在正常逻辑流中使用异常控制流程

### 3.4 副作用

- 函数的副作用必须在名称或文档中体现（如 `SaveUser` 明显有 IO 操作）
- 纯计算函数应无副作用，便于测试和复用

## 4. 注释规范

### 4.1 注释原则

- **注释解释"为什么"，代码解释"是什么"**
- 好的代码应自文档化，避免将注释作为烂代码的掩护
- 注释与代码同步更新，过期注释比没有注释更有害

### 4.2 必须注释的场景

- 非显而易见的业务逻辑或算法
- 临时性 workaround（须附带 TODO 和 issue 链接）
- 复杂的正则表达式
- 魔法数字和魔法字符串（应优先用常量替代，注释解释含义）

### 4.3 禁止注释的场景

- 简单明了的代码（`// 遍历用户列表` 等废话注释）
- 被注释掉的死代码（直接删除，版本控制会保留历史）
- 重复代码意图的注释（`i++ // i 加 1`）

### 4.4 TODO / FIXME 约定

```
TODO(<owner>): <描述> [<issue-link>]
FIXME(<owner>): <描述> [<issue-link>]
```

- `TODO`：已知待做项，不紧急
- `FIXME`：已知问题，需尽快修复
- 所有 TODO/FIXME 必须有 owner，建议附 issue 链接

## 5. 错误处理

### 5.1 错误处理原则

- 错误必须被处理，不允许静默忽略（空 catch / `_` 赋值需有注释说明原因）
- 错误在边界处（HTTP handler、队列消费者入口等）统一记录日志
- 业务错误与系统错误分类处理，向用户暴露友好错误信息

### 5.2 错误信息规范

- 错误信息使用英文（便于全球搜索和工具处理）
- 错误信息不以大写开头，不以标点结尾（便于包装拼接）
- 错误信息应包含足够上下文，但不暴露敏感信息（栈帧、SQL 语句、密钥等）

### 5.3 错误码体系

> [团队填写] 定义错误码体系，例如：
>
> - 错误码格式：`{模块代码}{错误类型}{序号}`，如 `USR4001`（用户模块，客户端错误，序号1）
> - 错误码统一在 `docs/standards/error-codes.md` 维护
> - 4xx：客户端错误，5xx：服务端错误（参照 HTTP 语义）

## 6. 日志规范

### 6.1 日志级别

| 级别    | 使用场景                             |
| ------- | ------------------------------------ |
| `DEBUG` | 开发调试信息，生产环境默认关闭       |
| `INFO`  | 关键业务操作（用户登录、订单创建等） |
| `WARN`  | 非预期但可恢复的情况（重试、降级）   |
| `ERROR` | 需要人工介入的异常                   |
| `FATAL` | 程序无法继续运行（慎用）             |

### 6.2 日志格式

- 使用结构化日志（JSON），禁止自由格式字符串拼接
- 必须字段：`timestamp`、`level`、`service`、`trace_id`、`message`
- 禁止在日志中记录：密码、Token、完整信用卡号、身份证号等敏感信息

### 6.3 日志内容原则

- 日志应能独立回答 "发生了什么、在哪里发生、上下文是什么"
- 每条 ERROR 日志应能触发对应的告警规则
- 高频路径（如每次 HTTP 请求）使用 DEBUG 级别，避免日志爆炸

## 7. 配置管理

### 7.1 配置原则（12-Factor App）

- 配置与代码分离：所有环境差异化配置通过环境变量或配置文件注入
- 不在代码仓库中提交任何密钥、密码、Token（包括测试环境）
- 配置项必须有默认值（开发友好）和校验逻辑（启动失败而非运行时失败）

### 7.2 密钥管理

> [团队填写] 例如：
>
> - 使用 Vault / AWS Secrets Manager / 环境变量注入
> - 本地开发使用 `.env.local`（加入 `.gitignore`）
> - 绝不将 `.env` 文件提交到仓库

### 7.3 配置文件命名约定

> [团队填写] 例如：
>
> - `config.yaml`（默认配置）
> - `config.<env>.yaml`（环境覆盖，如 `config.production.yaml`）

## 8. 代码 Review 检查清单

在 Review 他人代码或自我审查时，逐项确认：

### 正确性

- [ ] 是否覆盖了边界情况（空值、零值、最大值、并发）
- [ ] 错误是否被正确处理，而非静默忽略
- [ ] 是否存在资源泄露（连接、文件句柄、goroutine）

### 可读性

- [ ] 函数和变量命名是否清晰表达意图
- [ ] 复杂逻辑是否有必要的注释
- [ ] 是否有冗余或重复代码

### 安全性

- [ ] 用户输入是否经过验证和转义
- [ ] 是否有 SQL 注入、XSS、SSRF 等风险
- [ ] 日志是否泄露敏感信息

### 性能

- [ ] 是否有 N+1 查询
- [ ] 循环内是否有不必要的数据库/网络调用
- [ ] 是否需要考虑缓存

### 测试

- [ ] 新功能是否有对应的单元测试
- [ ] 测试是否覆盖了边界情况和异常路径
