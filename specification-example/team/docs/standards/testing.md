# 测试规范

> 本文件定义团队级测试策略与规范，适用于所有项目和技术栈。

---

## 1. 测试策略

### 1.1 测试金字塔

```
          /  E2E  \          ← 少量，验证关键用户路径
         /----------\
        /Integration \       ← 适量，验证模块间协作
       /──────────────\
      /   Unit Tests   \     ← 大量，验证单一逻辑单元
     /──────────────────\
```

| 类型     | 占比（建议） | 速度   | 关注点             |
| -------- | ------------ | ------ | ------------------ |
| 单元测试 | ~70%         | 毫秒级 | 函数/方法的正确性  |
| 集成测试 | ~20%         | 秒级   | 模块间、服务间协作 |
| E2E 测试 | ~10%         | 分钟级 | 完整用户场景       |

### 1.2 覆盖率要求

> [团队填写] 例如：
>
> - 新增代码行覆盖率不低于 80%（CI 门禁）
> - 核心业务逻辑（支付、权限）不低于 90%
> - 覆盖率仅作参考，不应为追求覆盖率写无价值的测试

---

## 2. 单元测试规范

### 2.1 测试范围

- 测试的单元：单个函数/方法，隔离外部依赖
- 需要单元测试的场景：业务逻辑、算法、工具函数、边界条件
- 不需要单元测试的场景：简单 getter/setter、框架本身的功能、配置文件

### 2.2 测试命名

测试名称应清晰表达：**测什么 + 在什么条件下 + 期望什么结果**

```
// 格式：<被测单元>_<条件>_<期望结果>
// 示例：
test("calculateDiscount_whenPremiumUser_returns20Percent")
test("getUser_whenUserNotFound_returnsNotFoundError")
test("parseDate_withInvalidFormat_throwsValidationError")
```

### 2.3 测试结构（AAA 模式）

```
// Arrange：准备数据和依赖
// Act：执行被测行为
// Assert：验证结果
```

每个测试只验证一个行为，不在同一测试中混合多个断言场景。

### 2.4 测试独立性

- 测试之间不共享状态，每个测试独立可运行
- 禁止测试依赖执行顺序
- 测试完成后清理所有副作用（数据库记录、文件、外部调用）

### 2.5 Mock 原则

- 只 mock 外部依赖（数据库、HTTP、消息队列），不 mock 内部业务逻辑
- Mock 对象应尽量简单，避免 mock 的行为比被测代码更复杂
- 使用 interface/协议 而非具体实现，便于 mock

---

## 3. 集成测试规范

### 3.1 集成测试范围

- 数据库访问层（Repository）与真实数据库的集成
- 多个服务模块协作的关键路径
- 外部 API 调用（使用测试环境或合同测试）

### 3.2 测试数据库

> [团队填写] 例如：
>
> - 使用 Docker 启动测试数据库实例（通过 testcontainers 管理）
> - 每个测试套件使用独立的 database / schema，避免干扰
> - 测试完成后自动清理数据

### 3.3 合同测试

> [团队填写] 如果有微服务间 API 合同测试需求，例如：
>
> - 使用 Pact 进行消费者驱动的合同测试
> - Provider 测试在 CI 中验证，确保合同未被破坏

---

## 4. E2E 测试规范

### 4.1 E2E 测试范围

- 只覆盖核心用户路径（注册、登录、核心业务流程）
- 不用 E2E 测试所有功能，成本高且维护难

### 4.2 测试环境

> [团队填写] 例如：
>
> - E2E 测试跑在独立的 staging 环境
> - 测试账号和数据由 fixture 脚本预置
> - 使用 Playwright 驱动浏览器

### 4.3 稳定性要求

- E2E 测试必须幂等（可重复运行，结果一致）
- 禁止 E2E 测试依赖外部可变状态（如第三方服务的真实数据）
- Flaky test 须立即修复或隔离，不允许长期存在

---

## 5. CI 测试要求

### 5.1 必须在 CI 中执行

- [ ] 单元测试（每次 PR）
- [ ] 集成测试（每次 PR 或 merge to main）
- [ ] 覆盖率报告生成和门禁检查
- [ ] E2E 测试（merge to main 或定时）

### 5.2 测试失败处理

- 任何测试失败必须阻塞合并（不允许 skip 或 force merge）
- 已知 flaky test 使用 quarantine 标记，并限时修复

> [团队填写] quarantine 标记方式：
>
> ```
> // @quarantine: JIRA-1234, 预期修复日期 2025-XX-XX
> ```

---

## 6. 测试数据管理

### 6.1 Fixture 原则

- 测试数据最小化：只创建测试所需的最少数据
- 使用工厂函数（Factory）生成测试对象，便于复用和维护
- 敏感数据（手机号、身份证等）使用脱敏的假数据

### 6.2 快照测试

- 快照测试（Snapshot Testing）仅用于 UI 组件，不用于业务逻辑
- 快照文件需要 review，不应盲目更新
- 快照文件提交到仓库

---

## 7. 测试反模式（禁止）

- 测试测试框架本身的功能（如验证 mock 是否被调用了正确次数——这是验证行为而非结果）
- 在测试中使用 `sleep` / `wait` 等时间依赖（使用确定性的异步控制）
- 为追求覆盖率写无实际价值的测试
- 在生产代码中插入仅供测试的逻辑（如 `if testing.mode`）
- 测试逻辑比被测代码更复杂
