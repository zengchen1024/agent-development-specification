# 文档写作规范

> 本文件定义代码库内文档（README、API 文档、注释、ADR 等）的写作规范。

---

## 1. 文档原则

- **对齐代码**：文档与代码同步更新，过期文档比没有文档更有害
- **读者视角**：为"第一次接触这个项目的工程师"而写
- **简洁优先**：用最少的字传达最多的信息，避免废话和重复
- **可操作性**：文档应包含足够的细节让读者能独立完成任务

---

## 2. README 规范

### 2.1 必须包含的章节

每个项目的 `README.md` 必须包含：

```markdown
# 项目名称

一句话描述这个项目是做什么的。

## 快速开始（Quick Start）

## 项目结构

## 开发指南

## 部署

## 贡献指南（可选，小项目可省略）
```

### 2.2 快速开始要求

- 从零开始到运行起来的完整步骤
- 包含所有前置依赖（版本号）
- 包含环境变量配置说明（`.env.example` 列举所有必填项）
- 步骤可验证（每一步都有预期输出或验证方式）

````markdown
## 快速开始

**前置依赖：**

- Go 1.22+
- Docker 24+
- make

**运行步骤：**

1. 克隆并安装依赖
   ```bash
   git clone <repo> && cd <project>
   make deps
   ```
````

2. 配置环境变量

   ```bash
   cp .env.example .env.local
   # 编辑 .env.local，填入必要配置
   ```

3. 启动服务
   ```bash
   make dev
   # 访问 http://localhost:8080/health 验证
   ```

````

### 2.3 写作风格

- 标题使用动词短语（`部署服务` 而非 `部署`）
- 命令行示例使用代码块，注明语言
- 说明步骤时编号（1、2、3），非步骤列表用 `-`

## 3. API 文档规范

### 3.1 文档格式

> [团队填写] 选择 API 文档规范，例如：
> - HTTP API：使用 OpenAPI 3.0（Swagger）规范，`openapi.yaml` 放在项目根目录
> - gRPC API：Proto 文件即文档，字段必须有注释
> - 内部文档补充：`docs/api/` 目录存放使用示例和业务说明

### 3.2 每个 API 端点必须描述

- 功能说明（一句话）
- 请求参数（类型、是否必填、取值范围）
- 响应结构（成功和失败）
- 错误码列表
- 调用示例（curl 或代码示例）

### 3.3 变更记录

- Breaking Change 必须在 CHANGELOG 中记录，并标注影响范围
- API 废弃须提前一个版本标注 `@deprecated`，并说明替代方案

## 4. 代码注释规范

> 详见 `coding/base.md` 第 4 节（注释规范）

**补充约定：**

- 公开函数/方法必须有文档注释（doc comment），描述：功能、参数语义、返回值、异常情况
- 包级别（package/module）注释说明整个包的职责
- 内联注释（行尾）解释非显而易见的逻辑，不重复代码意图

## 5. 架构决策记录（ADR）

### 5.1 什么时候写 ADR

- 技术选型（框架、数据库、中间件）
- 架构风格变更
- 影响多个团队或服务的设计决策
- 未来可能被质疑或推翻的决策（记录当时的背景和约束）

### 5.2 ADR 模板

```markdown
# ADR-NNNN: <标题>

**日期：** YYYY-MM-DD
**状态：** 提议 / 已接受 / 已废弃 / 被 ADR-XXXX 取代

## 背景

描述问题或需求背景，以及促使这次决策的因素。

## 决策

我们决定采用 <方案>，原因是...

## 考虑过的选项

| 选项 | 优点 | 缺点 |
|------|------|------|
| 方案 A | ... | ... |
| 方案 B | ... | ... |

## 后果

**正面影响：** ...
**负面影响 / 权衡：** ...
**需要跟进的行动：** ...
````

### 5.3 ADR 存储位置

- `docs/adr/0000-template.md`（模板）
- `docs/adr/NNNN-<kebab-case-title>.md`（序号递增）
- ADR 一旦接受，不修改内容；若决策改变，写新 ADR 并在旧 ADR 中标注状态

## 6. CHANGELOG 规范

### 6.1 格式

遵循 [Keep a Changelog](https://keepachangelog.com) 格式：

```markdown
# Changelog

## [Unreleased]

## [1.2.0] - 2025-06-01

### Added

- 新增用户批量导入功能

### Changed

- 优化列表查询性能

### Fixed

- 修复登录状态不持久的问题

### Deprecated

- `GET /api/v1/users/list` 将在 v2.0 删除，请使用 `GET /api/v2/users`

### Breaking Changes

- `createUser` API 参数 `mobile` 重命名为 `phone`
```

### 6.2 更新时机

- 每个合入 main 的 PR 同步更新 `[Unreleased]` 节
- 发版时将 `[Unreleased]` 更名为版本号和日期

## 7. Runbook / 操作手册

> [团队填写] 运维相关操作文档要求，例如：

> - 每个服务必须有 `docs/runbook.md`
> - 必须包含：服务简介、监控看板链接、常见故障处理、数据库操作注意事项、联系人
> - 格式参见 `docs/templates/runbook-template.md`
